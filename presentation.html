Playwright & SharedWorkers

Defeating the Chromium Sandbox

An architectural deep dive into worker network interception

The Goal 🎯

Selectively mock specific network (fetch) requests inside a SharedWorker during Playwright E2E testing.

...while allowing all other requests to perfectly hit the real backend server.

The Initial Problem 🚧

Playwright's page.route() seamlessly intercepts the Main Thread and Dedicated WebWorkers.

But it cannot see network traffic inside SharedWorkers.

Why? Because SharedWorkers run in a global multi-context architecture outside of a single page's direct control.

Attempt 1: Native Script Patching

If we can't intercept the worker's network requests, let's intercept the worker's source code download!


// Playwright intercepts the script download via CDP
await context.route('**/shared-worker.js', async (route) => {
  
  // Inject a monkey-patch mid-air
  const patch = `
    self.fetch = async (url) => {
      if (url === '/api/snapshot') return MOCK_RESPONSE;
      return originalFetch(url);
    };
  `;
  
  // Serve the tampered file to the browser
  return route.fulfill({ body: patch + originalScript });
});

The Result: CORS Blocked ❌

The mocked endpoint succeeded perfectly, but all real network requests instantly failed.

TypeError: Failed to fetch

Why? The Chromium Opaque Origin Sandbox!

The Interception & Sandbox 🕵️

  • The DevTools Protocol: Playwright intercepts and modifies the script "mid-air" using the Chrome DevTools Protocol.
  • What is a Sandbox? A sandbox is a heavily restricted execution environment inside the browser designed to isolate potentially malicious or untrusted code.
  • The Trigger: Because the script was artificially modified by DevTools (instead of natively downloading from your local server), Chromium's security engine gets suspicious and permanently traps the worker in this sandbox.

What is an "Opaque Origin"? 🛑

  • Normal Origins: Scripts usually inherit the trusted identity of the web page (e.g., http://localhost:8080).
  • Opaque (Null) Origins: When sandboxed, Chromium strips this identity away, assigning the worker an "Opaque Origin" (literally the string value null). It is now completely anonymous and untrusted.
  • The Consequence: When this anonymous worker tries to make a real API request, the browser's CORS (Cross-Origin) security blocks it. The backend server requires requests to come from a trusted origin, but the browser refuses to send the request because the origin is null.

Attempt 2: The Final Solution ✨

Blob Proxy Injection

How do we defeat the sandbox with zero changes to the application code?

What is a Blob? 🧩

  • Definition: Blob stands for Binary Large Object. In JavaScript, it is a way to store raw data (like a string of Javascript code) dynamically in the browser's memory.
  • Blob URLs: The browser can convert a Blob in memory into a temporary, local URL using URL.createObjectURL(blob). It looks like this:
    blob:http://localhost:8080/xyz-123
  • The Superpower: Because a Blob URL is generated directly by your Main Thread's Javascript, it permanently inherits the Main Thread's trusted origin (http://localhost:8080).

How the Blob Proxy Works 🛠️

  1. Intercept Instantiation: Playwright silently overrides the new SharedWorker() constructor in the Main Thread.
  2. Fetch as Text: The Main Thread downloads the patched worker script over the network as a raw text string.
  3. Wrap in a Blob: The Main Thread wraps that raw text into a Blob, and generates a trusted Blob URL.
  4. Execute: The worker boots from the trusted Blob URL instead of the suspicious network file!

Why the Blob Succeeds 🏆

Because the Blob URL inherits the Main Thread's exact identity, Chromium never sandboxes it.

No DevTools tampering flags.

No Opaque (Null) origin.

No CORS errors.

Conclusion 🎉

  • Targeted requests are perfectly mocked.
  • Real requests safely pass through to the backend.
  • The application architecture remains 100% untouched.
